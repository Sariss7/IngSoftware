Ciclos de Vida del Desarrollo del Software

El ciclo de vida del software permite planificar y organizar los procedimientos necesarios para su desarrollo, dando origen a diferentes metodologías. Estas metodologías guían los pasos para crear un producto de software, cada una diseñada para facilitar el desarrollo de proyectos específicos. Conocer el ciclo de vida ayuda a controlar mejor sus fases, evitando situaciones innecesarias. Las fases comunes incluyen una fase inicial, un proceso de desarrollo y un producto final. Aunque existen diversas metodologías, algunas, como el modelo de cascada, se consideran básicas y sirven como base para entender otros modelos de desarrollo de software.

1. Planificación.
La fase de planificación en el desarrollo de software incluye tareas como el análisis de costos y beneficios, la programación, la estimación de recursos y su asignación. En esta fase, el equipo de desarrollo reúne los requisitos de diversas partes interesadas (clientes, expertos y directivos) y crea un documento de especificaciones que define los objetivos del software. Este documento guía la planificación del proyecto, ayudando a estimar costos, establecer una programación y desarrollar un plan detallado para alcanzar los objetivos establecidos.

2. Fase de análisis.
En la fase de análisis de un proyecto se definen las funciones y características específicas que tendrá el software. Esto es fundamental para optimizar la asignación de costos y determinar el alcance del proyecto. La asignación de costos es una tarea compleja, ya que se realiza al inicio cuando el conocimiento sobre el proyecto es limitado, aumentando el margen de error. Sin embargo, la experiencia en proyectos similares y dividir el proyecto en tareas facilita una estimación más precisa del presupuesto.

3. Diseño.
Los desarrolladores usan la especificación de requisitos de software para diseñar la mejor arquitectura del producto. Proponen varias opciones de arquitectura basadas en estos requisitos y documentan las propuestas en el DDS (Design Document Specification). Las partes interesadas revisan el DDS, considerando aspectos como riesgos de mercado, modularidad, limitaciones de presupuesto y tiempo, y la solidez del producto, para seleccionar el mejor diseño. El diseño elegido detalla la arquitectura del producto en diagramas de flujo de datos, documentando cada aspecto, tanto interno como externo, en un documento de diseño completo..

4. Fase de desarrollo de un software.
Una vez definida la estructura del software, comienza la programación. Es crucial elegir un lenguaje adecuado y contar con programadores expertos. Seguir buenas prácticas de codificación facilita la escalabilidad del proyecto.

5. Mantenimiento.
En esta fase, el equipo corrige errores, resuelve problemas de clientes y gestiona cambios en el software. También supervisa el rendimiento, la seguridad y la experiencia del usuario para identificar mejoras.


Modelo de Ciclo de Vida de un Software
Los modelos proporcionan una metodología para validar todas las partes de un proyecto. A continuación, se presentan los distintos modelos utilizados en el desarrollo y preparación de software.


Modelo en Cascada.
Es un método de gestión que divide el proyecto en fases secuenciales lineales, donde cada fase debe completarse antes de iniciar la siguiente. Este modelo es simple, organizado y evita que las fases se mezclen.


Modelo en Iterativo.
Este proceso comienza con un subconjunto de requisitos y mejora el software en versiones sucesivas hasta que esté listo para producción. Al final de cada iteración, el equipo genera una nueva versión del software.


Modelo de Espiral.
Este modelo combina los ciclos repetidos del modelo iterativo con el enfoque secuencial del modelo en cascada, poniendo énfasis en el análisis de riesgos. Permite la actualización y mejora gradual del software mediante la creación de prototipos en cada fase.


Métodos agiles.
Son métodos de gestión de proyectos más flexibles, centrados en la interacción constante con el cliente. Sus características incluyen:
• Entregas frecuentes y revisión del trabajo por etapas.
• Adaptación a cambios basados en la retroalimentación del cliente.
• Reuniones periódicas para evaluar avances y resultados.
• Mejora continua del proyecto.


Métodos o Metodología Computacional.

Los modelos genéricos no son descripciones definitivas de procesos de software; son abstracciones útiles que pueden ser utilizadas para explicar diferentes enfoques del desarrollo de software (Pons- Giandini, y otros, 2010).


Sommerville define modelo de proceso de software como “Una representación simplificada de un proceso de software, representada desde una perspectiva específica. Por su naturaleza los modelos son simplificados, por lo tanto, un modelo de procesos del software es una abstracción de un proceso real (Sommerville, 2005).


Algunos de los modelos más conocidos son:
1. Prototipo.
2. Desarrollo basado en componentes (reutilización).
3. Desarrollo en espiral.
4. Modelo RAD (Rapid Application Development).
5. Modelo en cascada.


Prototipo.

Su objetivo es entender los requisitos del usuario y trabajar para mejorar la calidad de los mismos. Este modelo inicia con la recolección de requerimientos del cliente, con base en estos se define el conjunto de objetivos para el software, se identifican los requisitos conocidos y con base en estos se desarrolla rápidamente un prototipo o maqueta que posteriormente evalúa el cliente utilizándolo y ayudando a refinar de nuevo los requisitos del software a desarrollar; este proceso se seguirá repitiendo hasta que el cliente quede satisfecho con el desarrollo del software (Salazar-Aguirre, y otros, 2011).


Desarrollo basado en componentes (reutilización).

1. Los compromisos en los requisitos son inevitables, por lo cual puede que el software no cumpla las expectativas del cliente.
2. Las actualizaciones de los componentes adquiridos no están en manos de los desarrolladores del sistema.
3. Alto costo, si la adaptación de nuevos paquetes es costosa.
4. Dificultades para asegurar el versionamiento si los suministradores de componentes no mejoran continuamente sus productos.


Desarrollo en Espiral.

El modelo de desarrollo en espiral es actualmente uno de los más conocidos y fue propuesto por Barry Boehm en 1986. El ciclo de desarrollo se representa como una espiral, en lugar de una serie de actividades sucesivas con retrospectiva de una actividad a otra. Este enfoque entrelaza las actividades de especificación, desarrollo y validación. Es decir, surge de un sistema inicial que se desarrolla rápidamente a partir de especificaciones abstractas.


Modelo RAD.

Desarrollo Rápido de Aplicaciones (por sus siglas en inglés) es un modelo de proceso de desarrollo de software relativamente corto (dura entre 60 y 90 días). Se utiliza la construcción de software basada en componentes, utilizando herramientas de software que permitan de forma ágil y efectiva realizar una aplicación con altos estándares de calidad.


Modelo en Cascada.

Este modelo toma las actividades fundamentales del proceso de especificación, desarrollo, validación y evolución y las representa como fases separadas del proceso.

Proceso de Desarrollo de Software

Un proceso de desarrollo de software es el conjunto estructurado de las actividades requeridas para construir un sistema. El proceso de desarrollo de software se utiliza para mejorar la comprensión del problema a resolver, la comunicación entre los participantes del proyecto y el mantenimiento de un sistema complejo.


Existen herramientas para semiautomatizar o automatizar diferentes métodos del desarrollo de software, tales como el análisis, diseño, implementación o pruebas. Como ejemplos de herramientas clásicas para el desarrollo de software, podemos mencionar desde las iniciales herramientas CASE (del inglés, Computer-Aided software Engineering), hasta los más sofisticados ambientes integrados de desarrollo (del inglés, Integrated Development Environment).


Tanto las herramientas CASE como los IDE son aplicaciones informáticas, cuyo objetivo es proporcionar apoyo semiautomatizado o completamente automatizado en cualquiera de los métodos o actividades involucrados en el proceso de desarrollo de software. Las herramientas CASE y los IDE contribuyen significativamente a agilizar el proceso de desarrollo de software, reduciendo el costo del mismo en términos de tiempo, recursos humanos y recursos financieros.

¿Qué son los requerimientos?

Los requerimientos especifican qué es lo que un sistema de software debe hacer (sus funciones) y sus propiedades esenciales y deseables. Un requerimiento expresa el propósito del sistema sin considerar cómo se va a implantar. En otras palabras, los requerimientos identifican qué hace el sistema, mientras que el diseño establece el cómo lo hace el sistema.

Los requerimientos pueden servir a tres propósitos:

• Primero, permiten que los desarrolladores expliquen cómo han entendido lo que el cliente espera del sistema.

• Segundo, indican a los desarrolladores qué funcionalidad y qué características debe tener el sistema resultante.

• Tercero, indican qué demostraciones se deben llevar a cabo para convencer al cliente de que el sistema que se le entrega es de hecho lo que había ordenado.


La IEEE divide los requerimientos en funcionales y no funcionales:
Los requerimientos funcionales de un sistema describen lo que el sistema debe hacer.
Los requerimientos no funcionales ponen límites y restricciones al sistema.


Características:

• Deben ser correctos: Tanto el cliente como el desarrollador deben revisarlos para asegurar que no contengan errores.

• Deben ser consistentes: Dos requerimientos son inconsistentes cuando es imposible satisfacerlos simultáneamente.

• Deben ser realistas: Todos los requerimientos deben ser revisados para asegurar que sea posible implementarlos en un tiempo razonable.

• Deben ser verificables: Se deben poder preparar pruebas que demuestren que se han cumplido los requerimientos.


El proceso de requerimientos es el conjunto de técnicas y procedimientos que nos permiten conocer los elementos necesarios para definir un proyecto de software y, es una de las fases más importantes para el éxito del desarrollo del proyecto.

• Captura.
• Análisis.
• Especificación.
• Validación y verificación.
